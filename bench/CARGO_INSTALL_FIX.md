## Technical Notes

### Why bat/fd auto-detect but hyperfine does not

Auto-detect looks for registry paths containing `/src/main.rs` or `/src/bin/`:
- **bat**: `bat-0.26.1/src/bin/bat/main.rs` has panics → `/src/bin/` fires
- **fd-find**: `fd-find-10.4.2/src/main.rs:344` has a panic → `/src/main.rs` fires
- **hyperfine**: all embedded paths are `src/benchmark/...`, `src/timer/...` etc — no binary-entry-point file has an embedded panic in this version

The stem-matching fallback (binary name → crate name) would find `hyperfine` since binary="hyperfine"=crate, but the fallback only runs when the main-signal step finds zero or multiple candidates. Since hyperfine has zero main-signal candidates, the stem-matching step does run — but with zero dep-registry paths at all matching the stem (bug: actually the stem match runs before checking whether crate dirs exist, and returns Fallback if no registry crates are present). This is correct: the fallback correctly prints a message rather than silently producing wrong results.

### Why DWARF precision differs between local and registry builds

Local builds (git HEAD) are validated against their own DWARF, but the version mismatch means DWARF may map few user functions (e.g., bat local: only 70 DWARF-user fns mapped in a 5353-FDE binary → low DWARF precision). Registry builds use the SAME binary as both the test input and the DWARF GT twin (stripped vs unstripped of the same build), so DWARF precision is more reliable.

### Dep-isolation invariant

The critical invariant "dep crate paths must still classify Dep" holds for all fixtures. With `--crate bat`, the 212 dep-crate source strings (51 distinct deps: anyhow, tokio, syntect, etc.) all retain `Origin::Dep`. Only the 15 `bat-0.26.1/...` source strings are promoted to `Origin::User`. The xref scan consequently builds the correct dep_boundary set from dep Location structs.
