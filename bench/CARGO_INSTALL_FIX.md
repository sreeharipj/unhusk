# Cargo-Install Registry Fix — Validation Results

Generated by `bench/run_cargo_install.sh`.  
Fixtures built with `CARGO_PROFILE_RELEASE_DEBUG=2 CARGO_PROFILE_RELEASE_STRIP=false cargo install --root /tmp/ci-test`.

**The bug (pre-fix):** cargo-install binaries embed all source paths as registry paths
(`~/.cargo/registry/src/<hash>/<crate>-<ver>/`). The original `classify_path` mapped every
registry path to `Dep`, so n_certain=0 for all 52 cargo-install binaries in the sweep.

**The fix:** `--crate <name>` promotes that crate's registry paths to `User`. Auto-detect
infers the root crate from embedded paths (main/bin signal, then binary-stem fallback).

## Technical Notes

### Auto-detect signal coverage across fixtures

All three auto-detect paths are exercised:

| Signal | Fires for | Mechanism |
|--------|-----------|-----------|
| `/src/main.rs` panic | fd-find | `fd-find-10.4.2/src/main.rs:344` has a panic site |
| `/src/bin/` panic | bat | `bat-0.26.1/src/bin/bat/main.rs` has panic sites |
| Stem match | hyperfine | No entry-point panics; binary stem `"hyperfine"` matches sole registry crate dir `hyperfine-1.20.0` |

For the stem-match path to work, the binary must be invoked (or the fixture named) with the
installed binary's actual name, not a test-rig suffix. The harness preserves this by using
`<binname>.stripped` where `file_stem()` returns `binname`.

### DWARF validator fix (9b37a83)

`dwarf::read_function_sources` now accepts `root_crates` and passes them to the DWARF path
classifier. Without this, the DWARF ground truth for cargo-install binaries also classified
the root crate's source paths as `Dep` (since DWARF paths are also registry paths), making
DWARF precision meaningless. With the fix, DWARF agrees with the symbol GT (e.g., hyperfine
DWARF precision = 90.9% = symbol precision).

### Local vs registry DWARF precision discrepancy (bat)

bat local build shows DWARF precision 8.9% vs registry 100%. The local build uses git HEAD
(different version); DWARF maps only 70 of 5353 functions as user-first-party. This is a
version-mismatch artifact, not a classification error. Symbol precision (nm-based) is the
reliable metric across versions: 99.25% local vs 99.18% registry, Δ = 0.07pp.
